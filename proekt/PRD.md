## 2. Контекст и проблема

### 2.1 Нормативная база

В соответствии со статьёй 219 НК РФ физические лица, уплатившие НДФЛ по ставке 13%, вправе получить социальный налоговый вычет за приобретение медикаментов, назначенных лечащим врачом. Предельный размер вычета — **150 000 руб. в год** (с 2024 года). Вычет подаётся за 3 предыдущих года.

**Обязательные документы для ИФНС:**
- Декларация 3-НДФЛ
- Справка 2-НДФЛ от работодателя
- Документ, подтверждающий назначение препарата врачом (рецепт или альтернатива — см. ниже)
- Кассовые чеки / платёжные документы
- Заявление на возврат налога

> ℹ️ С 2019 года штамп «Для налоговых органов» и ИНН на рецепте **не обязательны** — рецепт без штампа не является основанием для отказа (письмо ФНС №БС-3-11/937@ от 10.02.2020).

### 2.2 Допустимые документы вместо рецепта

Согласно письму ФНС **№СД-4-11/11080@ от 24.08.2022**:

| Документ | Форма | Когда применяется | Принимает ФНС |
|---|---|---|---|
| Рецепт врача | 107-1/у | Амбулаторное лечение | ✅ Основной |
| Электронный рецепт | ЕГИСЗ | Любое | ✅ |
| Выписка из медкарты амбулаторного пациента | 025/у | Амбулатория, нет рецепта | ⚠️ Спорно |
| Медкарта стационарного больного | 003/у | Стационар | ✅ |
| Медкарта стоматологического больного | 043/у | Стоматология | ✅ |
| Карта беременной и родильницы | 111/у | Беременность | ✅ |
| Талон пациента | 025-1/у | Амбулатория | ⚠️ Спорно |
| Консультативный лист | — | — | ❌ Не принимается |

> ⚠️ Выписки форм 025/у и 025-1/у (амбулатория) инспекторы трактуют по-разному. Основной путь — рецепт 107-1/у.

### 2.3 Боли пользователей

- Чеки теряются, выцветают, накапливаются за год
- Рецепты хранятся отдельно, привязка к покупкам — ручной труд
- Группировка по месяцам для 3-НДФЛ занимает часы
- Нет единого инструмента: люди ведут Excel или фотографируют в галерею
- Заполнение 3-НДФЛ без бухгалтера вызывает затруднения

## 3. Архитектура OCR-пайплайна

> ⚠️ **Текущая реализация**: только библиотеки EasyOCR + Tesseract. ФНС Open API и Google Vision API — отдельные задачи модернизации (Appendix A.9).

### 3.1 Два источника чеков

| Источник | Метод | Точность | Когда используется |
|---|---|---|---|
| QR-код → декодирование | pyzbar + OpenCV | ~95% (данные из QR) | QR читается |
| EasyOCR (основной) | PyTorch CRNN | 85–92% | Нет QR или плохое фото |
| Tesseract 5 (fallback) | LSTM OCR | 80–86% | EasyOCR упал / таймаут |
| 🔮 ФНС Open API [будущее] | HTTP → nalog.ru | ~100% | После регистрации юрлица |
| 🔮 Google Vision API [будущее] | Cloud API | 96–99% | Pro-тариф, Фаза 7+ |

### 3.2 Сравнение бесплатных OCR-библиотек

| Библиотека | Лицензия | Русский | Вес | CPU скорость | Точность на чеках |
|---|---|---|---|---|---|
| **EasyOCR ✓ выбран** | Apache 2.0 | ✅ Хорошо | ~200 МБ | Медленно | 85–92% |
| PaddleOCR | Apache 2.0 | ✅ Хорошо | ~10 МБ | Быстро | 88–94% |
| Tesseract 5 (fallback) | Apache 2.0 | ✅ Хорошо | ~50 МБ | Быстро | 80–88% |
| DocTR | Apache 2.0 | ⚠️ Ограничено | ~100 МБ | Средне | 82–89% |

> PaddleOCR не выбран: тащит экосистему Baidu PaddlePaddle — тяжёлая зависимость для продакшна.

### 3.3 Параллельный пайплайн v2

```
Фото чека → QR decode (sync, < 0.5 сек)
                │
                ├─ QR есть + чек < 12 мес:
                │     ├── [async] QR-данные обрабатываются ────────┐
                │     └── [thread] EasyOCR ──────────────────────── ResultMerger
                │                                                   │
                └─ нет QR / чек старый:                            │
                      └── [thread] EasyOCR ────────────────────────┘
```

| | v1 (последовательный) | v2 (параллельный) |
|---|---|---|
| Время обработки | до 12+ сек | **3–6 сек** |
| Кросс-проверка | Нет | **ResultMerger** |

### 3.4 Логика давности чека (ReceiptAgeEstimator)

| Возраст чека | Стратегия |
|---|---|
| 0–12 месяцев | EasyOCR + QR-декодирование |
| > 12 месяцев | Только OCR (фото чека) |
| Чек утерян | Запрос дубликата у аптеки (54-ФЗ, 5 лет) |

### 3.5 ResultMerger — 6 стратегий слияния

| Стратегия | Условие | Confidence | Статус |
|---|---|---|---|
| `merged` | Суммы и даты совпали | 1.0 | DONE (авто) |
| `merged_date_conflict` | Суммы совпали, дата расходится | 0.92 | DONE (авто) |
| `fns_only` | Только QR/ФНС сработал | из источника | DONE / REVIEW |
| `ocr_only` | Нет QR или чек старый | из OCR | DONE / REVIEW |
| `conflict` | Суммы расходятся (> 2% и > 5 руб.) | ≤ 0.75 | REVIEW |
| `both_failed` | Оба источника не дали результат | 0.0 | FAILED |

### 3.6 Предобработка изображений (OpenCV)

- **Deskew** — исправление наклона (мин. < 0.5° игнорируется)
- **CLAHE** — адаптивное выравнивание контраста по регионам
- **Adaptive Threshold** — бинаризация при неравномерном освещении
- **Fast Non-Local Means Denoising** — удаление зернистости
- **Auto-upscale** — масштабирование если ширина < 1200px

| Сценарий фото | Основной путь | Ожид. точность | Статус |
|---|---|---|---|
| Чек с QR | QR decode | ~95% | DONE |
| Чёткое фото, хорошее освещение | EasyOCR | 88–92% | DONE |
| Среднее фото (тени, наклон) | Предобработка + EasyOCR | 80–88% | DONE / REVIEW |
| Плохое фото / выцветший чек | Tesseract fallback | 70–82% | REVIEW |
| Нечитаемое изображение | — | < 60% | FAILED |

## 7. Алгоритм обработки одного чека

### 7.1 Шаги обработки

| № | Шаг | Действие системы | Результат |
|---|---|---|---|
| 1 | Получение фото | Бот принимает фото, сохраняет в YOS, создаёт `receipts` со статусом PENDING | «Принял, обрабатываю...» |
| 2 | Параллельный OCR | QR-декодирование + EasyOCR одновременно | `ParsedReceipt` |
| 3 | ResultMerger | Слияние результатов по 6 стратегиям | `confidence` + `merge_strategy` |
| 4 | Нормализация препаратов | Fuzzy match против ГРЛС | `drug_inn` + `is_rx` |
| 5 | Поиск рецепта в БД | Для каждого `is_rx=true`: запрос к `prescriptions` по МНН + дата | Рецепт найден / не найден |
| 6а | Рецепт найден | Показать реквизиты, предложить привязать | Привязать / Другой рецепт |
| 6б | Рецепт не найден | Уведомить. Предложить 3 варианта | Загрузить / Ввести / Позже |
| 7 | Проверка confidence | ≥ 0.85 → DONE; 0.60–0.84 → REVIEW; < 0.60 → FAILED | Уведомление с подсказкой |
| 8 | Сохранение в БД | `receipts` + `receipt_items` + `prescription_id` | Финальный статус |

### 7.2 Алгоритм поиска рецепта (L1–L4)

| Уровень | Критерий | SQL-условие | Что показывать |
|---|---|---|---|
| **L1** | Точное совпадение МНН + дата в пределах срока | `drug_inn = МНН AND purchase_date BETWEEN issue_date AND expires_at` | ✅ Найден — предложить привязать |
| **L2** | МНН совпал, рецепт просрочен ≤ 30 дней | `drug_inn = МНН AND purchase_date - expires_at ≤ 30` | ⚠️ Просрочен на N дней |
| **L3** | Fuzzy по торговому названию (score ≥ 85) | `rapidfuzz(drug_name, prescription.drug_name) ≥ 85` | ⚠️ Возможно подходит — проверьте |
| **L4** | Ничего не найдено | — | ❌ Загрузить / Ввести / Позже |

### 7.3 Сценарий «Рецепт не найден»

| Выбор | Действие системы | Следующий шаг |
|---|---|---|
| 📷 Загрузить фото | Бот ждёт фото → OCR рецепта → нераспознанные поля выделяются | Форма с предзаполненными полями → привязка |
| ✏️ Ввести вручную | ConversationHandler: врач → специальность → клиника → дата → препарат → доза → срок | Сохранение → привязка |
| 🕐 Позже | Чек сохраняется с `needs_prescription=true` | Напоминание через 3 дня. Без рецепта — не попадёт в пакет ИФНС |

### 7.4 Ручная корректировка OCR рецепта

| Поле рецепта | OCR уверенность | Поведение |
|---|---|---|
| ФИО врача | Часто низкая (рукопись) | Всегда запрашивать вручную при conf < 0.7 |
| Дата выписки | Средняя (цифры читаются) | Автозаполнение + подтверждение |
| Название препарата | Низкая (латиница) | Fuzzy ГРЛС → топ-3 варианта кнопками |
| Дозировка | Средняя | Автозаполнение + подтверждение |
| Специальность врача | Средняя | Автозаполнение или список специальностей |
| Наименование клиники | Высокая (печать) | Автозаполнение без подтверждения |

## 10. Аутентификация и защита ПД

### 10.1 Схема входа по номеру телефона через Telegram

| № | Шаг | Детали |
|---|---|---|
| 1 | /start в боте | Кнопка «Поделиться контактом» — Telegram передаёт верифицированный номер |
| 2 | Создание аккаунта | `users`: `telegram_id` + `phone_hash` (bcrypt) + `telegram_username` |
| 3 | Вход на сайт | Пользователь вводит номер → «Получить код» |
| 4 | OTP | Система генерирует 6-значный код (TTL 5 мин) → отправляет через Telegram-бот |
| 5 | Проверка кода | Пользователь вводит код → JWT access + refresh |
| 6 | Сессия | access TTL 15 мин, refresh TTL 30 дней с rotation |
| 7 | Mini App | Автоматический вход через `initData` (HMAC-SHA256) |

> ✅ Telegram-сообщение бесплатно, мгновенно, не зависит от оператора связи.

### 10.2 Хранение персональных данных

> ⚠️ Медицинские данные (сведения о покупке лекарств, рецепты) — **специальная категория ПД по ст. 10 152-ФЗ**.

| Данные | Способ хранения | Меры защиты |
|---|---|---|
| Номер телефона | PostgreSQL: bcrypt-хеш | Нельзя восстановить |
| ФИО, ИНН, СНИЛС | PostgreSQL: AES-256 на уровне приложения | Ключ в Vault / env, не в БД |
| Фото чеков | YOS: приватный бакет, AES-256 | pre-signed URL TTL 15 мин |
| Фото рецептов / медкарт | YOS: отдельный приватный бакет | Дополнительная проверка владельца |
| OCR-результаты (препараты) | PostgreSQL: открытый текст | RLS изоляция. TLS 1.3 при передаче |
| JWT-токены | httpOnly cookie | Защита от XSS |
| Логи | Sentry (анонимизированные) | ПД в логах маскируются |

### 10.3 Обязательные документы ПД

| Документ | Где в интерфейсе | Обязательность |
|---|---|---|
| Политика обработки ПД (152-ФЗ) | Подвал сайта, /start в боте, раздел «Профиль» | ✅ До запуска |

### 13.1 Производительность

- Аудит-трейл всех обращений к ПД

- Stateless API-серверы за балансировщиком
- Celery воркеры масштабируются независимо от API

## A.5 SSE-протокол

**Эндпоинт:** `GET /api/v1/batch/{batch_id}/stream`
**Авторизация:** обязательна. `Content-Type: text/event-stream`
**Архитектура:** FastAPI `StreamingResponse` + Redis PubSub (канал: `batch:{batch_id}`)

| event_type | data поля | Когда |
|---|---|---|
| `batch_started` | `batch_id, total_files, source` | После создания `batch_job` |
| `item_classified` | `item_id, filename, classified_as` | После классификации каждого файла |
| `item_done` | `item_id, receipt_id, confidence, pharmacy_name, purchase_date, total_amount` | confidence ≥ 0.85 |
| `item_review` | `item_id, receipt_id, confidence, user_hint, low_confidence_fields: list` | confidence 0.60–0.84 |
| `item_failed` | `item_id, error_code, user_hint` | confidence < 0.60 |
| `prescription_linked` | `prescription_id, receipt_item_id, match_level` | Автопривязка из пачки |
| `batch_complete` | `batch_id, done, review, failed, duration_sec` | Все файлы обработаны |
| `heartbeat` | `timestamp` | Каждые 15 сек (предотвращение таймаута) |

**Формат сообщения:**
```
event: item_done
data: {"item_id": "uuid", "receipt_id": "uuid", "confidence": 0.92, ...}
id: 42

```

## A.6 PostgreSQL RLS

**Файл:** `backend/alembic/versions/0004_rls_policies.py`

Приложение устанавливает `app.current_user_id` перед каждым запросом через middleware (`app/middleware/rls.py`):

```python
async with db.begin():
    await db.execute(
        text("SET LOCAL app.current_user_id = :uid"),
        {"uid": str(user.id)}
    )
```

| Таблица | Политика |
|---|---|
| `receipts` | `USING (user_id = current_setting('app.current_user_id')::uuid)` |
| `receipt_items` | `EXISTS (SELECT 1 FROM receipts r WHERE r.id = receipt_id AND r.user_id = current_setting(...)::uuid)` |
| `prescriptions` | `USING (user_id = current_setting('app.current_user_id')::uuid)` |
| `batch_jobs` | `USING (user_id = current_setting('app.current_user_id')::uuid)` |
| `otp_codes` | RLS не применяется — нет `user_id` |

> ⚠️ **Celery-воркеры** работают без пользовательского контекста. Роль БД `medvychet_worker` должна иметь `BYPASS ROW LEVEL SECURITY`.

## A.8 Классификатор batch

**Файл:** `backend/app/services/ocr/batch_classifier.py`
Вызывается на Шаге 1 пакетной обработки **до** полного OCR.

| Шаг | Проверка | Результат | Уверенность |
|---|---|---|---|
| 1 | Декодировать QR (pyzbar). Параметры `fn/i/fp/t/s` присутствуют | `receipt` | 1.0 — однозначно чек |
| 2 | Первые 5 строк Tesseract (`--psm 6`). Слова: «ФИСКАЛЬНЫЙ», «ФН», «ФД», «ФП», «АПТЕКА», «ИТОГО» | `receipt` | 0.85 если 3+ слова |
| 3 | Слова рецепта: «Rp.», «Рецепт», «107-1/у», «МНН», «Дозировка», «Врач» | `prescription` | 0.85 если 2+ слова |
| 4 | Слова медкарты: «ВЫПИСКА», «МЕДИЦИНСКАЯ КАРТА», «003/у», «025/у», «ДИАГНОЗ» | `prescription` | 0.80 если 2+ слова |
| 5 | Ничего не подошло | `unknown` | 0.0 — ручная классификация |

**Возвращаемый объект:**
```python
ClassificationResult(
    classified_as: str,          # receipt | prescription | unknown
    confidence: float,
    keywords_found: list[str],
    recommended_doc_type: DocType | None
)
```

> 💡 **Оптимизация:** классификация использует только первые 10 строк Tesseract (`psm 6`) — занимает ~0.3 сек. Полный EasyOCR запускается зная тип файла, что позволяет использовать разные постпроцессоры для чеков и рецептов.
